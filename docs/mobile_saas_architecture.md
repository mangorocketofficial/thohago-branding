# Thohago Mobile SaaS Architecture

> Status: Draft  
> Date: 2026-04-08  
> Branch: `mobile-web-pivot`  
> Scope: 모바일 웹앱 기반 콘텐츠 생성 SaaS 아키텍처 재정의

---

## 1. 문서 목적

이 문서는 `Thohago_branding_mobile` 프로젝트를 기준으로, 지금까지 논의한 모바일 SaaS 방향을 하나의 설계 문서로 정리한다.

핵심 재정의는 다음과 같다.

- 고객은 모바일 웹앱/PWA에서만 서비스를 사용한다.
- 고객은 사진/영상 업로드와 인터뷰만 진행한다.
- 서버가 정해진 파이프라인으로 결과물을 생성한다.
- 고객은 완성 파일을 다운로드해서 직접 플랫폼에 업로드한다.
- 자유 편집은 제공하지 않는다.
- 결과가 마음에 들지 않으면 재생성만 가능하다.

이 문서는 제품 정의, UX, 과금, 백엔드 구조, 세션 구조, 기술 스택, 구현 원칙을 포함한다.

---

## 2. 제품 정의

### 2.1 제품 한 줄 정의

`Thohago는 모바일 웹앱에서 사진/영상 업로드와 인터뷰를 받은 뒤, 서버가 콘텐츠를 생성하고 고객이 완성본을 다운로드하는 모바일 SaaS이다.`

### 2.2 이 제품이 해결하려는 문제

소상공인 사용자는:

- 휴대폰에 이미 사진/영상이 있다.
- 긴 편집 과정을 원하지 않는다.
- AI 툴을 직접 세밀하게 다루고 싶지 않다.
- 다만 “바로 올릴 수 있는 결과물”은 원한다.

기존 방향의 문제는 다음과 같았다.

- 데스크톱 편집기 방향:
  - 직접 편집을 해야 차별점이 생기는데, 그렇게 가면 기존 편집툴과 크게 다르지 않다.
- 완전 대행 방향:
  - 사람이 개별 결과물을 계속 손보게 되면 운영 공수가 급격히 증가한다.

따라서 최종 방향은:

- 모바일 입력 UX
- 서버 통제형 생성 파이프라인
- 수동 업로드용 완성본 패키지 전달

으로 수렴한다.

### 2.3 제품 포지션

이 제품은 아래 둘 중 어느 것도 아니다.

- 자유 편집형 AI 편집툴
- 사람 손이 계속 들어가는 콘텐츠 대행 서비스

대신 아래에 가깝다.

- `모바일 입력 + 서버 생성 + 결과 다운로드` 구조의 콘텐츠 생성 SaaS

## 3. 핵심 제품 원칙

### 원칙 1. 사용자는 편집하지 않는다

- 자연어 편집
- 세부 레이아웃 수정
- 수동 타임라인 편집

은 제공하지 않는다.

사용자는:

- 업로드
- 인터뷰
- 결과 선택
- 재생성

만 수행한다.

### 원칙 2. 품질은 사람 편집이 아니라 범위 제약으로 확보한다

품질을 높이기 위해 운영자가 매번 결과를 손보지 않는다.

대신:

- 입력 형식 제한
- 생성 포맷 제한
- 업종별 패턴화
- 결과물 구조 고정

으로 변동성을 줄인다.

### 원칙 3. 비용, 모델, 파이프라인은 서버가 통제한다

고객이 Gemini API 키를 직접 붙이지 않는다.

서버가:

- 모델 선택
- 프롬프트 관리
- 렌더링
- 비용 추적

을 전부 통제한다.

### 원칙 4. 사용자는 필요한 결과물만 생성한다

4종 전체를 무조건 결제시키지 않는다.

사용자는:

- 블로그만
- 영상만
- 캐러셀만
- 스레드만

선택해서 생성할 수 있어야 한다.

### 원칙 5. UI는 자유 채팅이 아니라 워크플로 채팅이다

UI는 채팅처럼 보이지만 실제로는 상태 머신 기반이다.

- 자유 메시지는 인터뷰 답변에만 허용
- 나머지는 버튼, 카드, 업로드, 다운로드 중심

## 4. 사용자 경험 구조

## 4.1 접근 방식

서비스 진입점은 모바일 웹앱이다.

흐름:

1. 사용자가 모바일 웹페이지 접속
2. 구글 로그인
3. 결제
4. 바로 작업 시작
5. 필요하면 홈 화면에 추가하여 PWA처럼 사용

PWA 설치는 별도 앱 설치 파일 개념이 아니다.

- 홈 화면 아이콘 생성
- 브라우저 주소창 없이 앱처럼 실행

정도로 이해하면 된다.

## 4.2 로그인과 PWA 설치의 관계

권장 흐름:

- 웹에서 로그인/결제/서비스 진입을 먼저 완료
- 이후 사용자가 원하면 홈 화면에 추가

비권장 흐름:

- 결제 후 PWA 설치
- PWA에서 다시 로그인

즉 PWA는 별도 서비스가 아니라 동일 웹서비스의 설치형 진입점이다.

## 4.3 채팅형 UX 원칙

사용자 UI는 단일 채팅 화면으로 구성한다.

단:

- ChatGPT처럼 여러 대화방을 만들지 않는다.
- 하나의 메인 채팅 타임라인만 존재한다.
- 새 작업을 시작할 때 채팅 안에 새 세션 블록을 생성한다.

### 채팅 안에 존재하는 요소

- 시스템 메시지
- 사용자 인터뷰 답변 메시지
- 업로드 카드
- 상태 카드
- 생성 결과 카드
- 다운로드 카드
- 재생성 버튼 카드

### 허용되는 사용자 자유 입력

- 인터뷰 답변만 자유 입력

### 허용되지 않는 것

- 결과 수정 요청 자유 채팅
- 자연어 기반 편집
- 일반 상담 챗봇 형태의 자유 대화

즉 UI는:

- `자유 채팅 UI`가 아니라
- `워크플로를 채팅처럼 보이게 만든 UI`

이다.

## 5. 모바일 UX 상세 플로우

## 5.1 최초 진입

1. 랜딩 페이지
2. 구글 로그인
3. 요금제 확인 / 결제
4. “새 작업 시작” 버튼

## 5.2 새 작업 시작

채팅 안에 새 세션 블록이 생성된다.

예시:

- 시스템: “새 작업을 시작할게요. 사진과 영상을 업로드해 주세요.”
- 카드: 사진/영상 업로드

## 5.3 업로드 단계

사용자는:

- 사진 업로드
- 영상 업로드

를 모바일에서 수행한다.

업로드 완료 후 서버는:

- 미디어 저장
- media preflight
- 대표 사진 후보 선정

을 수행한다.

## 5.4 인터뷰 단계

시스템은 카드 기반으로 질문을 제시한다.

입력 방식:

- 음성 녹음
- 텍스트 입력

인터뷰는 3턴 구조를 유지하되:

- Turn 1
- Turn 2
- Turn 3

질문은 서버에서 생성한다.

## 5.5 생성 선택 단계

인터뷰가 끝나면 시스템이 생성 가능한 콘텐츠 카드를 보여준다.

예:

- 블로그 생성
- 캐러셀 생성
- 영상 생성
- 스레드 생성

사용자는 필요한 타입만 선택한다.

## 5.6 결과 전달 단계

서버는 결과를 완성 파일로 만든다.

사용자에게는 채팅 카드로 전달한다.

예:

- 블로그:
  - 제목
  - 본문
  - 해시태그
  - 복사 버튼
  - 다운로드 버튼
- 캐러셀:
  - 이미지 ZIP
  - 캡션 TXT
  - 다운로드 버튼
- 릴스/쇼츠:
  - MP4
  - 캡션 TXT
  - 썸네일 후보
- 스레드:
  - 복사용 본문 카드

## 5.7 재생성 단계

사용자가 결과가 마음에 들지 않으면:

- 다시 생성
- 톤 변경 재생성
- 짧게
- 길게
- 더 프리미엄하게
- CTA 강화

같은 bounded regeneration만 실행할 수 있다.

직접 편집은 없다.

## 6. 세션 구조

## 6.1 사용자와 채팅 관계

사용자마다 하나의 메인 채팅 워크스페이스가 있다.

하지만 백엔드 내부적으로는 작업 세션이 여러 개일 수 있다.

예:

- 사용자 A
  - 세션 1: 봄 이벤트 블로그
  - 세션 2: 릴스 영상
  - 세션 3: 캐러셀 재생성

UI는 하나지만 데이터는 세션별로 분리 저장한다.

## 6.2 권장 데이터 모델

- `users`
- `billing_accounts`
- `usage_ledger`
- `chat_threads`
- `work_sessions`
- `messages`
- `media_files`
- `session_artifacts`
- `generation_requests`
- `generation_results`
- `download_packages`

## 6.3 메시지 구조

메시지는 크게 다음 타입으로 관리한다.

- `system_text`
- `user_text`
- `user_audio`
- `upload_card`
- `status_card`
- `generation_card`
- `download_card`

## 7. 과금 모델

## 7.1 권장 과금 방식

`사용량 기반 요금제`

단, 사용자에게는 토큰/원가를 직접 드러내지 않는다.

표면상은:

- 사용량
- 포인트
- 크레딧

형태로 보여준다.

## 7.2 왜 사용량 기반이 맞는가

4종 전체 세트 강제는 비합리적이다.

사용자는:

- 블로그만 필요할 수 있음
- 영상만 필요할 수 있음
- 재생성을 많이 돌릴 수 있음

따라서 결과물 유형별 차등 차감이 맞다.

## 7.3 내부 사용량 계산 기준

내부 계산은 아래를 조합한다.

- 외부 API 사용량
- 콘텐츠 유형별 가중치
- 렌더링 비용
- 재생성 여부

예:

- 블로그 생성: 낮은 가중치
- 캐러셀 생성: 중간 가중치
- 스레드 생성: 낮은 가중치
- 영상 생성: 높은 가중치

## 7.4 사용자 노출 방식

예시:

- 블로그 생성: 8포인트
- 캐러셀 생성: 10포인트
- 스레드 생성: 6포인트
- 영상 생성: 20포인트
- 재생성: 최초 생성보다 낮은 포인트 차감

## 7.5 요금제 예시

- Starter: 월 100포인트
- Growth: 월 300포인트
- Pro: 월 800포인트

또는:

- 기본 월 구독
- 사용량 초과 시 상위 요금제 업그레이드

## 8. 백엔드 아키텍처

## 8.1 기본 구조

권장 구조는 다음과 같다.

- API 서버
- 비동기 작업 워커
- 파일 저장소
- 데이터베이스

### API 서버 역할

- 로그인/세션
- 업로드 처리
- 인터뷰 상태 관리
- 생성 요청 접수
- 사용량 차감
- 결과 다운로드 URL 반환

### 작업 워커 역할

- Vision preflight
- 인터뷰 질문 생성
- 블로그/캐러셀/영상/스레드 spec 생성
- PIL 렌더링
- FFmpeg 렌더링
- 결과 패키징

### 파일 저장소 역할

- 원본 업로드 파일
- 중간 아티팩트
- 최종 결과물
- 다운로드 패키지 ZIP

## 8.2 처리 방식

API 요청-응답 안에서 영상 렌더링을 직접 하지 않는다.

권장:

- API 서버가 job 생성
- 워커가 비동기로 처리
- 모바일은 polling 또는 SSE로 상태를 받음

예:

- `POST /sessions/{id}/generate/video`
- 응답: `job_id`
- `GET /jobs/{id}`
- 완료 후 `download_url` 반환

## 8.3 서버에서 직접 만드는 결과물

서버는 최종 산출물을 완성형태로 만든다.

예:

- 블로그 markdown/txt
- 캐러셀 이미지 세트
- MP4 영상
- 스레드 텍스트
- ZIP 패키지

즉 모바일은 얇고, 서버가 무겁다.

---

## 9. 인터뷰 및 미디어 파이프라인

## 9.1 핵심 원칙

웹 버전에서 중요했던 구조를 유지한다.

- 사진/영상 업로드 후 먼저 `media_preflight` 생성
- preflight를 바탕으로 질문 생성
- 인터뷰 답변과 preflight를 함께 콘텐츠 생성에 사용

## 9.2 media preflight

서버는 업로드 직후 multimodal vision 호출을 수행한다.

저장 대상:

- `media_preflight.json`
- DB 세션 preflight 필드

포함 정보 예시:

- structure_mode
- key_visual_evidence
- question_focus_candidates
- photo_annotations
  - scene
  - details
  - mood
  - selected_for_prompt

## 9.3 질문 생성

권장 구조:

- Turn 1:
  - 고정 질문 또는 preflight 기반 질문
- Turn 2:
  - preflight + Turn 1 답변 기반 생성
- Turn 3:
  - preflight + Turn 1/2 답변 기반 생성

중요:

- desktop처럼 단순 템플릿 기반으로 끝내지 않는다.
- Gemini 기반의 동적 질문 생성이 필요하다.

## 9.4 콘텐츠 생성

콘텐츠별로 독립 요청 가능해야 한다.

- 블로그
- 캐러셀
- 영상
- 스레드

사용자는 4종 전체를 강제 생성하지 않는다.

---

## 10. 결과물 전달 구조

## 10.1 결과물은 직접 업로드용 패키지로 제공

퍼블리싱은 서버가 직접 수행하지 않는다.

서버는:

- 결과물을 생성
- 다운로드 가능한 형태로 제공

사용자는:

- 직접 플랫폼에 업로드

## 10.2 콘텐츠별 전달 형식

### 블로그

- 제목
- 본문
- 해시태그
- 복사 버튼
- markdown/txt 다운로드

### 캐러셀

- 이미지 묶음
- 캡션
- ZIP 다운로드

### 영상

- MP4
- 캡션 텍스트
- 필요 시 썸네일 후보

### 스레드

- 복사 가능한 본문 카드
- txt 다운로드

## 11. 기술 스택 선택

## 11.1 현재 코드베이스 상태

`Thohago_branding_mobile`의 현재 웹 스택은 이미 다음 구조를 갖고 있다.

- FastAPI
- Jinja2
- static CSS
- 바닐라 JS
- SSE
- PWA(manifest + service worker)

즉 현재는 React/Vite 기반이 아니다.

## 11.2 권장 선택

현 시점에서는 현재 스택을 유지하는 쪽이 유리하다.

권장 스택:

- 백엔드/API: FastAPI
- 템플릿/UI: Jinja2
- 프론트 인터랙션: 바닐라 JS
- 실시간 상태 전달: SSE
- PWA: manifest + service worker

## 11.3 왜 이 스택이 맞는가

현재 제품은:

- 복잡한 편집기 아님
- 업로드/인터뷰/상태/다운로드 중심
- 선형 워크플로
- 모바일 중심

이런 특성상:

- React/Vite SPA보다
- 서버 렌더링 + 최소 JS

가 더 빠르고 가볍고 단순하다.

## 11.4 React/Vite를 도입해야 하는 경우

아래가 생기면 그때 재검토한다.

- 복잡한 클라이언트 상태 공유
- 자유 편집기
- 고급 인터랙션이 많은 화면
- 대규모 오프라인 캐싱

현재는 필요하지 않다.

## 12. PWA 전략

## 12.1 PWA의 의미

PWA는 별도 설치 파일을 내려받는 앱이 아니다.

사용자가:

- 모바일 웹에 접속하고
- 홈 화면에 추가하면
- 앱 아이콘처럼 실행할 수 있게 되는 웹앱

이다.

## 12.2 사용자 체감

- 홈 화면 아이콘 존재
- 주소창 없이 앱처럼 실행
- 하지만 본질은 웹앱

## 12.3 제품 전략

PWA는 서비스의 별도 버전이 아니다.

권장 흐름:

- 웹에서 로그인
- 웹에서 결제
- 웹에서 바로 사용
- 원하면 홈 화면에 추가

즉 설치는 선택 옵션이다.

- 사용량 대시보드
- 추천 생성 흐름

## 16. 최종 결론

`Thohago_branding_mobile`은 모바일 웹/PWA 기반의 콘텐츠 생성 SaaS로 재정의한다.

고객은:

- 모바일에서 업로드
- 인터뷰
- 생성 요청
- 결과 다운로드

를 수행한다.

서버는:

- vision preflight
- 질문 생성
- 콘텐츠 생성
- 렌더링
- 패키징

을 담당한다.

이 구조는:

- 데스크톱 편집기보다 사용자 입력 UX가 좋고
- 대행 서비스보다 운영 공수가 낮으며
- 현재 코드베이스(FastAPI + Jinja2 + PWA)와도 잘 맞는다.
