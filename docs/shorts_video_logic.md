# 쇼츠 영상 생성 핵심 로직

## 1. 문서 목적

이 문서는 뚜호고(Thohago) 서비스에서 쇼츠/릴스/숏폼 영상을 만들 때 필요한 핵심 백엔드 로직을 정리한 문서다.

핵심 전제는 다음과 같다.

- 블로그가 가장 풍부한 원본 서사다
- 쇼츠는 블로그의 서사를 영상 문법에 맞게 압축한 결과물이다
- 영상 편집 툴 자체보다, 어떤 훅과 감정 곡선으로 영상을 구성할지 결정하는 상위 기획 레이어가 중요하다

즉, 쇼츠는 "새로 만드는 콘텐츠"가 아니라 "블로그의 서사를 숏폼 문법으로 번역한 콘텐츠"로 본다.

---

## 2. 핵심 원칙

### 2.1 블로그 우선 원칙

- 블로그는 `master narrative` 역할을 한다
- 사진 순서, 장면 의미, 고객 반응, 차별점, 맥락 정보는 블로그에서 가장 잘 정리된다
- 쇼츠는 블로그 내용을 그대로 줄이는 것이 아니라, 블로그의 핵심 story unit들을 추출해 짧은 감정 곡선으로 재배열한다

### 2.2 편집 실행과 편집 기획 분리

영상 제작은 두 층으로 분리되어야 한다.

1. 편집 기획 레이어
- 어떤 훅으로 시작할지
- 어떤 감정 곡선을 탈지
- 어떤 장면을 어떤 순서로 배치할지
- 어떤 메시지를 남기고 무엇을 버릴지

2. 편집 실행 레이어
- 영상 자르기
- 사진 Ken Burns
- 영상/사진 이어붙이기
- 자막/텍스트 오버레이
- 색감/전환/음악/출력 포맷

현재 시스템에서 편집 실행은 이미 어느 정도 가능하다. 앞으로의 핵심은 편집 기획 레이어를 구조화하는 것이다.

### 2.3 길이 제약 우선 원칙

- 쇼츠는 주어진 소스 길이를 모두 쓰는 것이 목적이 아니다
- 먼저 사용 가능한 총 길이를 계산하고, 그 길이에 맞는 감정 곡선 구조를 선택해야 한다
- 스크립트는 영상 길이에 맞춰 생성되는 것이 아니라, 먼저 배분된 beat duration 안에서 작성되어야 한다

---

## 3. 입력과 출력

### 3.1 입력

쇼츠 생성 로직의 입력은 다음과 같다.

- 최종 블로그 본문
- 블로그 생성에 사용된 story context
  - 메인 angle
  - 인터뷰 전사
  - 사진 설명
  - 영상 설명
- 실제 편집 가능한 소스
  - 원본 영상 n개
  - 사진 n장
- 소스별 길이 정보
  - 영상 길이
  - 사진 최소/최대 홀드 가능 시간

### 3.2 출력

쇼츠 생성 로직은 최종적으로 다음 산출물을 만든다.

- `blog_story_map.json`
- `shorts_beat_sheet.json`
- `shorts_render_spec.json`
- 최종 렌더링 결과물(mp4)

---

## 4. 전체 파이프라인

전체 흐름은 아래와 같다.

1. 블로그 생성
2. 블로그에서 `story map` 추출
3. 사용 가능한 총 길이 계산
4. 길이 구간(duration band) 결정
5. 쇼츠용 beat sheet 생성
6. 각 beat에 맞는 스크립트/자막 생성
7. 각 beat에 맞는 사진/영상 소스 매핑
8. 최종 render spec 생성
9. 렌더링 실행

즉,

`블로그 -> story map -> beat sheet -> render spec -> 영상`

구조로 가는 것이 핵심이다.

---

## 5. 블로그를 마스터 서사로 쓰는 방식

### 5.1 왜 블로그가 기준인가

- 블로그는 가장 많은 맥락을 담고 있다
- 인터뷰와 사진이 어떤 관계를 갖는지 이미 정리되어 있다
- 이미지 순서와 의미가 설명되어 있다
- 차별점/고객 반응/장소 맥락이 문장으로 정리되어 있다

따라서 쇼츠는 원본 인터뷰와 사진만 다시 보는 것이 아니라, 블로그를 통해 이미 해석된 구조를 바탕으로 압축해야 한다.

### 5.2 블로그는 요약 대상이 아니라 압축 대상이다

중요한 점은 "블로그를 줄인다"가 아니다.

- 단순 문장 요약: 안 됨
- 감정 곡선 중심 압축: 해야 함

예를 들어 블로그가 다음 순서로 서술되더라도:

- 방문 맥락
- 시술 시작
- 장비/분위기
- 고객 반응
- 마무리

쇼츠는 반드시 같은 순서를 따를 필요는 없다.

쇼츠는 오히려:

- Hook
- Curiosity
- Proof
- Reaction
- CTA

처럼 재배열될 수 있어야 한다.

즉, `블로그는 정보 구조`, `쇼츠는 감정 구조`다.

---

## 6. Story Map 계층

쇼츠 기획 전에 먼저 블로그를 구조화한 `story map`이 필요하다.

### 6.1 역할

`story map`은 블로그에서 어떤 unit들이 실제로 중요한지를 정리하는 레이어다.

예:

- 방문 맥락
- 왜 이 샵을 선택했는지
- 시술/장비 증거
- 고객 반응
- 마무리/CTA

### 6.2 예시 스키마

```json
{
  "main_angle": "필리핀 관광객 5명이 한국 오기 전부터 예약한 헤드스파 경험",
  "story_units": [
    {
      "id": "arrival_context",
      "type": "context",
      "importance": 0.85,
      "photo_ids": [1],
      "video_ids": [],
      "summary": "필리핀 관광객들이 한국 오기 전 미리 예약했다"
    },
    {
      "id": "treatment_proof",
      "type": "proof",
      "importance": 0.92,
      "photo_ids": [2, 4],
      "video_ids": [1],
      "summary": "두피 진단과 LED 장비를 활용한 관리"
    },
    {
      "id": "reaction_peak",
      "type": "reaction",
      "importance": 0.97,
      "photo_ids": [3, 5],
      "video_ids": [],
      "summary": "K-뷰티에 대한 강한 만족 반응"
    }
  ]
}
```

### 6.3 핵심 판단 기준

각 story unit은 최소한 아래를 가져야 한다.

- 정보 중요도
- 감정 강도
- 시각적으로 보여줄 수 있는지 여부
- 숏츠에서 살릴 가치

## 7. 길이 기반 표준화

### 7.1 왜 길이 표준화가 필요한가

클라이언트가 주는 소스는 매번 다르다.

예:

- 13초 영상 1개
- 7초 영상 1개
- 사진 5장

이런 경우 총 사용 가능 길이는 약 22~23초가 된다.

하지만 중요한 건 "총 길이"보다 "목표 길이 구간"이다.

### 7.2 길이 구간 정의

예시:

- `short`: 12~18초
- `standard`: 18~26초
- `extended`: 26~35초

시스템은 소스 총량을 보고 `target_output_sec`를 먼저 정해야 한다.

### 7.3 길이 구간별 감정 곡선 구조

#### short (12~18초)

- Hook
- Proof
- Reaction
- CTA

#### standard (18~26초)

- Hook
- Setup
- Proof
- Reaction
- CTA

#### extended (26~35초)

- Hook
- Setup
- Contrast
- Proof
- Reaction
- CTA

즉 길이가 길어질수록 중간 비트를 추가하고, 짧을수록 더 압축한다.

---

## 8. Beat Sheet 계층

### 8.1 역할

`beat sheet`는 실제 쇼츠 편집의 설계도다.

여기서 결정해야 하는 것:

- 훅 종류
- 비트 순서
- 각 비트 길이
- 각 비트의 감정 목적
- 각 비트에 들어갈 소스 타입
- 각 비트의 자막/스크립트 예산

### 8.2 예시 스키마

```json
{
  "duration_band": "standard",
  "target_duration_sec": 24,
  "editorial_pattern": "tourist_reaction",
  "emotion_curve": ["curiosity", "proof", "delight", "release"],
  "beats": [
    {
      "name": "hook",
      "goal": "낯선 맥락으로 시선 잡기",
      "duration_sec": 3,
      "script_chars_max": 18,
      "asset_rule": "가장 놀라운 장면"
    },
    {
      "name": "setup",
      "goal": "왜 이 경험이 특별한지 설명",
      "duration_sec": 4,
      "script_chars_max": 26,
      "asset_rule": "맥락을 보여주는 사진/영상"
    },
    {
      "name": "proof",
      "goal": "장비, 과정, 차별점 제시",
      "duration_sec": 6,
      "script_chars_max": 34,
      "asset_rule": "기술/장비가 보이는 컷"
    },
    {
      "name": "reaction",
      "goal": "고객 반응 또는 감정 피크",
      "duration_sec": 5,
      "script_chars_max": 28,
      "asset_rule": "표정, 리액션, 후기"
    },
    {
      "name": "cta",
      "goal": "부드러운 마무리",
      "duration_sec": 3,
      "script_chars_max": 18,
      "asset_rule": "브랜드/공간/결과 장면"
    }
  ]
}
```

## 9. 쇼츠 스크립트 생성 원칙

### 9.1 스크립트는 비트별로 생성

스크립트를 한 번에 생성하면 길이 제어가 어렵다.

따라서 반드시:

- 각 beat의 목적
- 각 beat의 길이
- 각 beat의 글자 수 제한

을 먼저 정하고, 그 뒤에 beat 단위로 작성해야 한다.

### 9.2 스크립트 생성 규칙

- Hook는 가장 강한 angle 1개만 사용
- Setup은 맥락만 짧게 설명
- Proof는 장비/장면/과정의 증거를 보여줌
- Reaction은 감정 피크를 잡음
- CTA는 세게 팔지 않고 자연스럽게 마무리

즉, 쇼츠 스크립트는 블로그 문장을 줄이는 것이 아니라 `비트별 메시지 슬롯`을 채우는 방식이어야 한다.

## 10. 사진과 영상의 역할 분리

### 10.1 영상 소스

영상은 감정의 흐름을 만들고 템포를 잡는 주재료다.

활용 방식:

- Hook 컷
- 실제 과정 증명
- 리액션 장면
- 움직임이 중요한 구간

### 10.2 사진 소스

사진은 길이 보정과 의미 압축에 강하다.

역할:

- 부족한 길이 채우기
- 템포 조절
- 장비/공간/결과 장면을 명확하게 보여주기
- 특정 비트를 더 선명하게 만들기

즉, 사진은 단순 보조가 아니라 `길이 보정 + 리듬 조절 + 증거 강화` 역할이다.

### 10.3 사진 위주 숏츠 vs 영상 위주 숏츠

소스 상황에 따라 분기해야 한다.

- `footage-rich`
  - 실제 영상 중심
- `photo-led`
  - 사진 중심
  - motion, text, voiceover를 더 정교하게 사용
- `hybrid`
  - 영상 + 사진 혼합

사진만 있는데 영상처럼 과하게 흉내내면 싸구려 느낌이 나기 쉽다. 이 경우는 차라리 더 느린 motion, 더 좋은 훅, 더 명확한 text 구조가 낫다.

## 11. 감정 곡선 설계

### 11.1 감정 하나를 중심으로 잡기

한 쇼츠는 감정 중심축이 하나여야 한다.

예:

- 놀람
- 편안함
- 프리미엄
- 함께하는 경험
- 한국적 특별함
- 전문성 신뢰

### 11.2 감정 곡선 예시

#### 관광객 반응형

- Hook: "한국 오기 전부터 왜 예약했을까?"
- Setup: "부산 관광 동선 중 선택"
- Proof: "스팀캡, LED, 두피 진단"
- Reaction: "K-뷰티 최고라는 반응"
- CTA: "서면 오면 쉬어가기 좋은 곳"

#### 조용한 프리미엄형

- Hook: "도심 속에서 조용히 쉬는 장면"
- Setup: "캔들, LED, 공간 분위기"
- Proof: "관리 장비와 루틴"
- Reaction: "몸이 풀리는 순간"
- CTA: "프리미엄 휴식 경험"

즉, 감정 곡선은 영상의 분위기를 정하고, 컷 선택과 자막 톤까지 좌우한다.

## 12. Editorial Pattern 라이브러리

### 12.1 왜 필요한가

LLM에게 매번 "센스 있게 만들어"라고 하면 흔들린다.

따라서 패턴을 미리 정의해두는 것이 중요하다.

예:

- `tourist_reaction`
- `group_visit_story`
- `first_time_kbeauty`
- `before_after_relief`
- `quiet_luxury_spa`
- `specialized_treatment`
- `local_hidden_gem`

### 12.2 각 패턴이 가져야 할 요소

- 훅 공식
- 감정 곡선
- 컷 템포
- 자막 밀도
- CTA 스타일
- 어울리는 소스 타입

LLM은 새로 창작하기보다, 이 패턴들 중 가장 맞는 것을 선택하고 빈칸을 채우는 역할을 해야 한다.

## 13. 최종 Render Spec 계층

`render spec`은 마지막 단계에서만 생성한다.

여기에는 다음이 들어간다.

- asset sequence
- clip in/out
- photo hold duration
- Ken Burns path
- transition type
- text overlay
- voiceover script
- subtitle timing
- BGM mood
- export settings

즉,

- `story map`은 의미 구조
- `beat sheet`는 편집 구조
- `render spec`은 실행 구조

이다.

---

## 14. 실제 구현 기준 권장 계층

최종적으로 백엔드는 아래 3단계 JSON을 만든다.

1. `blog_story_map.json`
- 블로그에서 추출한 핵심 story unit

2. `shorts_beat_sheet.json`
- 길이 기반 감정 곡선과 비트 배치

3. `shorts_render_spec.json`
- 실제 렌더링 명세

이 3단계 구조를 지켜야:

- 블로그와 쇼츠의 메시지가 일관되고
- 길이 제약을 통제할 수 있으며
- 영상 편집 센스를 반복 가능한 로직으로 만들 수 있다

---

## 15. 결론

쇼츠 영상 생성의 핵심은 편집 툴이 아니라 상위 기획 로직이다.

정리하면:

- 블로그를 `master narrative`로 사용한다
- 블로그에서 `story map`을 추출한다
- 소스 길이에 맞춰 `duration band`를 선택한다
- 감정 곡선 기반 `beat sheet`를 만든다
- 비트별 스크립트와 asset 배치를 결정한다
- 마지막에만 `render spec`으로 변환해 실행한다

즉, 쇼츠는 "블로그의 요약본"이 아니라:

`블로그 서사를 숏폼 감정 곡선으로 압축 재편집한 영상`

이어야 한다.
